### I-5 — MZ extraction wrote every entry to `outDir/<kind>` (path + data loss)

**File:** `fsharp/src/RpgmDecrypt.Core/Report.fs` (MZ branch)
**Severity:** important — silent data loss; only surfaced because the C-2 test drove `Report.run`.

`Path.Combine(safeDir, entryName) |> renameByKind kind` evaluates (F# pipe +
currying) as `renameByKind (relPath = kind) (kind = combinedPath)`, so the
non-matching `kind` fell to `| _ -> relPath` and returned the bare kind string.
Result: **every** `.pak` entry was written to `outDir/png` (or `/ogg`, …), each
overwriting the last — the entry's real path was discarded entirely. The shipped
MZ tests only exercised `Mz.decryptAll` directly, never the `Report.run` write
path, so this was invisible.

**Fix (applied):** `let entryOutRel = renameByKind (Path.Combine(safeDir, entryName)) kind`.
Locked by a new "MZ .pak preserves nested entry path" end-to-end test.

## ⚠ CRITICAL FINDINGS (2)

### C-1 — `the CLI module` DU-case typo breaks the CI build and silently mislabels MV/MZ in reports

**File:** `fsharp/src/RpgmDecrypt.Cli/the CLI module:7-8`
**Severity:** critical — build fails under documented CI configuration; silent output corruption otherwise.

The union case is `VXAce` (`Types.fs:9`). `the CLI module` writes it as `VxAce`:

```fsharp
let hrFmt =
    function XP -> "XP" | VX -> "VX" | VxAce -> "VxAce" | MV -> "MV" | MZ -> "MZ"
//                                    ^^^^^ not the DU case → wildcard variable binding
```

Because `VxAce` is not a known case, F# treats it as a *variable pattern* that matches **anything**. Consequences:

- **Compile warnings** `FS0049` (uppercase identifier in pattern) + `FS0026` (the `MV` and `MZ` rules "will never be matched"). The CI build step runs `dotnet build … -p:TreatWarningsAsErrors=true` (`ci.yml:44-47`), so these warnings become **errors** and the build fails. The whole CI pipeline (test + publish + release, all `needs: test`) never runs. `docs/PROJECT_OVERVIEW.md:7,153` claims "Release, 0 warnings" — actual plain build emits **6 warnings**.
- **Without** the flag (plain `dotnet build`), it compiles, but `hrFmt MV` → `"VxAce"`, `hrFmt MZ` → `"VxAce"`, `hrFmt VXAce` → `"VxAce"`. So `runSummaryToJson`/`runSummaryHuman` report an MV game's files under `"VxAce":N`. The primary stdout report is wrong for the two most common engines.

The correct copy already exists in `Log.fs:37` (`… | VXAce -> "VXAce" | …`) — the two `hrFmt`s have drifted (see M-1).

Note: `dotnet run --project fsharp/src/RpgmDecrypt.Tests` passes 83/83 because the Tests project references **Core only**, never compiling `the CLI module` — so the test suite cannot catch this.

**Reproducer:**
```bash
# build failure under CI flags:
dotnet build fsharp/src/RpgmDecrypt.slnx -c Release -p:TreatWarningsAsErrors=true
#   the CLI module(8,40): error FS0049 ; (8,59)/(8,72): error FS0026 ; Build FAILED.

# silent mislabel (plain build): run on any MV game → JSON shows "VxAce":N, never "MV":N
```

**Fix:**
```fsharp
// the CLI module — delete the local hrFmt entirely and reuse the library mapping:
let hrFmt = Format.toString            // Types.fs already has the correct total function
// (or, minimally, rename the case: ... | VXAce -> "VXAce" | ...)
```

### C-2 — Zip-Slip / path-traversal: archive entry names yield arbitrary file write

**File:** `fsharp/src/RpgmDecrypt.Core/Report.fs:68-76` (VX Ace), `Report.fs:130-135` (MZ)
**Severity:** critical — CWE-22. A crafted archive writes files outside `OutDir` when a victim unpacks it. This is the core function of the tool (unpacking third-party archives), so the input is inherently untrusted.

VX Ace path (entry name comes from the decrypted archive TOC):
```fsharp
let outputRel =
    e.Name.Replace('\\', Path.DirectorySeparatorChar)
       .Replace('/', Path.DirectorySeparatorChar)
let outputPath = Path.Combine(cfg.OutDir, outputRel)   // no containment check
...
File.WriteAllBytes(outputPath, plain)
```

MZ path (entry name = `ZipArchiveEntry.FullName`, attacker-controlled):
```fsharp
let entryOutRel = Path.Combine(safeDir, entryName) |> renameByKind kind
let entryOutAbs = Path.Combine(cfg.OutDir, entryOutRel)
writeAllBytes entryOutAbs bytes
```

`.NET Path.Combine` discards the first argument when the second is **absolute**: `Path.Combine("out", "C:\\Windows\\x")` → `"C:\\Windows\\x"`. And `..\..\` segments resolve above `OutDir`. So an entry named `..\..\..\Users\asad\AppData\Roaming\Microsoft\Windows\Start Menu\Programs\Startup\x.bat` (or an absolute path) is written verbatim. `writeAllBytes` even `CreateDirectory`s the parent first.

The audit-request summary claims a `safeOutputPath` guard on "ALL write paths" — **it does not exist** (`grep -rn safeOutputPath fsharp/src/` → nothing). XP/VX only emit a `.entries.txt` manifest under the walker-derived (trusted) `RelPath`, so they are not affected; only MZ and VX Ace extract entry-named payloads.

**Reproducer:**
```bash
# Build a .pak (ZIP) whose entry name is "../../evil.txt", run the tool on it:
#   python -c "import zipfile; z=zipfile.ZipFile('game.pak','w'); z.writestr('../../evil.txt', b'x'); z.close()"
#   rpgm-decrypt <dir-containing-game.pak> ./out --password 00000000000000000000000000000000
# → ./evil.txt is created two levels ABOVE ./out
```

**Fix:** containment-check every extracted path before writing:
```fsharp
let private safeJoin (outDir: string) (rel: string) : string option =
    let root = Path.GetFullPath outDir
    let full = Path.GetFullPath (Path.Combine(root, rel))
    let rootSep = if root.EndsWith (string Path.DirectorySeparatorChar) then root
                  else root + string Path.DirectorySeparatorChar
    if full = root || full.StartsWith(rootSep, StringComparison.Ordinal) then Some full else None
// reject (Skipped/Failed) any entry whose name escapes outDir; also strip leading "/" and drive roots.
```

## IMPORTANT FINDINGS (4)

### I-1 — `Format.Vx.fs` missing negative-`name_len` guard → unhandled crash; also silently returns partial TOC

**File:** `fsharp/src/RpgmDecrypt.Core/Format.Vx.fs:68-75`
**Severity:** important — crash (exit 1) on corrupt/crafted input; silent truncation otherwise. Zero test coverage for VX.

XP (`Format.Xp.fs:95`) and VX Ace (`Format.VxAce.fs:106`) guard with `nLenInt < 0 || pos + nLenInt > buf.Length`. VX does **not**:

```fsharp
if nameLen = 0u && size = 0u then keepGoing <- false
elif pos + int nameLen > buf.Length then          // int(0x80000000u) = -2147483648
    keepGoing <- false
    Error Truncated |> ignore                      // dead statement (see M-4)
else
    let nameBytes = Array.zeroCreate<byte> (int nameLen)   // negative count → throws
```

A `name_len` with the high bit set casts to a negative `int`; `pos + negative` is `< buf.Length`, so the guard is skipped and `Array.zeroCreate<byte> -N` throws `ArgumentOutOfRangeException`. `Vx.parse`/`parseFile` are not wrapped, so the exception escapes to `Report.run`'s `VX` arm (which, unlike `VXAce`, has no `try`) and aborts the run. This is exactly the bug `PROJECT_OVERVIEW.md:196` says was fixed for VX Ace in `ec286f1` — the fix was never mirrored to VX.

Separately, even on a benign truncation the VX branch does not `acc.Clear()` (XP does), so it returns `Ok(partialEntries, pos)` and signals success despite a truncated table — a silent-correctness divergence from XP.

**Reproducer:** `Vx.parse [|0x52;0x47;0x53;0x53;0x41;0x44;0x00;0x02; 0;0;0;0; 0;0;0;0xFF; …|]` (size≠0, name_len high-bit set) → `ArgumentOutOfRangeException`.

**Fix:** copy XP's guard verbatim into VX (`nameLenInt < 0 || pos + nameLenInt > buf.Length` → `acc.Clear(); keepGoing <- false`), and add a `FormatVxTests.fs` mirroring `FormatXpTests.fs`. Better: extract the shared XP/VX parser the request summary already imagines (`RgssadCore`) so the guard cannot drift again.

### I-2 — Every MV asset is decrypted twice

**File:** `fsharp/src/RpgmDecrypt.Core/Format.fs:57-66`
**Severity:** important — 2× CPU and 2× allocation on the hot path (thousands of files per game).

```fsharp
let out, kind = Mv.decryptBytes key bytes        // calls Mv.decrypt → full XOR pass + alloc
let actuallyDecrypted =
    match Mv.decrypt key bytes with              // SECOND full XOR pass + alloc, result discarded
    | Mv.Plaintext _ -> false | Mv.Decrypted _ -> true | Mv.Unsure _ -> true
```

`THEORY.md:176-178` advertises "pre-allocate `byte[]` arrays once per file" — the actual path allocates the plaintext buffer twice per file and re-XORs the whole buffer.

**Fix:** call `Mv.decrypt` once and derive both values from the single `DecryptOutcome`:
```fsharp
match Mv.decrypt key bytes with
| Mv.Plaintext(kind, b) -> Ok(b, kind, false)
| Mv.Decrypted(kind, b) -> Ok(b, kind, true)
| Mv.Unsure b           -> Ok(b, "bin", true)
```

### I-3 — Exit code 4 is overloaded and contradicts the docs

**File:** `fsharp/src/RpgmDecrypt.Cli/the CLI module:174,188`
**Severity:** important — breaks scripts that branch on exit code.

Code `4` is returned both for "no key recovered" (`:174`) and for "0 inputs scanned" (`:188`). `the CLI module:114` help text and `PROJECT_OVERVIEW.md:131` say `4 = no key`; `README.md:88` says `4 = no recognised files`. Three sources, two meanings, one code. Also `:188 if summary.InputsScanned = 0 then exitWith 4` fires even when a key *was* found but the directory simply had no assets.

**Fix:** give "no recognised files" its own code (or fold into `0`), and align all three docs with `the CLI module`.

### I-4 — CI release job can never succeed (no artifact is ever uploaded)

**File:** `.github/workflows/ci.yml:57-175`
**Severity:** important — tag releases are broken.

The three `cli-*` jobs `dotnet publish … -o artifacts/<rid>` but contain **no `actions/upload-artifact` step**. The `upload` job then does `actions/download-artifact@v4 with: name: cli-linux-x64` (etc.) — names that were never produced. `download-artifact` errors when the named artifact is absent, so every `v*` tag build fails at `upload`. (Moot today because `test` itself fails per C-1, but it must be fixed too.)

**Fix:** add `- uses: actions/upload-artifact@v4` (with matching `name:` and `path:`) to each publish job.

## MINOR FINDINGS (12)

### M-1 — Two `hrFmt` copies drifted; both duplicate `Format.toString`
`the CLI module:7` (broken, C-1) and `Log.fs:35-37` (correct) re-implement the mapping already in `Types.fs:14-20`. Collapse both call sites to `Format.toString` (mirror-skill: one source of truth).

### M-2 — Dead `safe` byte-clamp (ported from a signed-byte language)
`VxAceKey.fs:47,69`, `the CLI module:162`: `let safe (b: byte) = if int b < 0 then byte (256 + int b) else b`. F# `byte` is unsigned, so `int b` is always `0..255`; the branch is unreachable. Harmless but misleading — delete.

### M-3 — No-op match in the orchestrator
`Report.fs:103`: `renameByKind d.RelPath (match d.Format with MV -> "bin" | MZ -> "bin" | _ -> "bin")` — all arms return `"bin"`, and `renameByKind … "bin"` hits the `| _ -> relPath` identity branch. The whole expression is `outRel = d.RelPath`. Refactor leftover; simplify.

### M-4 — Dead `Error Truncated |> ignore` statements
`Xp.fs:101`, `Vx.fs:71`: an `Error` value is constructed and discarded inside the imperative loop; correct error signaling actually depends on `acc.Count`/`acc.Clear()`. Confusing and fragile — replace with a real `result`/sentinel.

### M-5 — Per-4-byte allocation in VX Ace stream cipher
`VxAceKey.fs:48-58,71-87`: `decodeFilename` rebuilds a 4-byte `keyBytes()` array on *every byte*; `decodePayload` rebuilds `kb` every 4 bytes. On a multi-MB payload that is hundreds of thousands of tiny allocations (GC churn) — and contradicts `THEORY.md §8`. Compute the four key bytes inline with shifts/masks.

### M-6 — Documentation drift (multiple)
- `THEORY.md:105-107` & `README.md:134`: VX Ace layout stated as `size | name_len | name | offset` — the real layout (and `Format.VxAce.fs:14-22`) is `offset | size | entry-key | name_len | name`. Doc is wrong.
- `THEORY.md:180`: "manual fill via `Mutables.fill` API in `Crypto.fs`" — no such API; it's `Crypto.zeroFill`.
- `THEORY.md:160-163`: claims outputs `_manifest.json` / `_report.txt` / `_failures.jsonl` — none are ever written (`Report.fs` emits the mirror tree + `*.entries.txt` only).
- `THEORY.md §6` / `README.md:96-98`: key-discovery regexes listed don't match `KeyDiscovery.fs:69-79`.
- `README.md:69-78`: options table omits `--vxace-seed` (exists in code). `README.md:151` says "7 test modules" (9 are registered in `Main.fs`; 11 test files). `README.md:51` "Native AOT binary" — CI publishes self-contained **single-file**, not AOT (`PublishSingleFile`, not `PublishAot`).

### M-7 — Test passes for the wrong reason (VX Ace negative-name_len regression)
`FormatVxAceTests.fs:143-163`: buffer is only `header(8)+16 = 24` bytes, so `parse` exits at `pos + 16 > buf.Length` (24) **before** reading any field; it never reaches the `nLenInt < 0` guard it claims to regression-test, and the bytes are written to the *offset* field, not name_len. It returns `Truncated` via the short-buffer path and asserts pass. The actual guard is untested. Build a ≥`28+`-byte buffer with valid offset/size and a high-bit name_len.

### M-8 — Coverage gaps
No `Format.Vx.fs` test at all (see I-1). No Zip-Slip/traversal test (C-2). No negative tests for `Crypto.decodeHexKey` (odd length / non-hex `invalidArg`) or `xorTransform` empty-key. No test compiles `the CLI module` (so C-1 is invisible to CI) — add a tiny CLI smoke that asserts `hrFmt MV = "MV"`.

### M-9 — CI hardening gaps

## Root-cause pattern

The defects cluster around **two missing single-sources-of-truth**:

1. **Format-name mapping** is copy-pasted three times (`Types.toString`, `Log.hrFmt`, `Program.hrFmt`); one copy drifted into C-1.
2. **The RGSSAD parser** was copy-pasted between `Xp.fs` and `Vx.fs`; the negative-`name_len` hardening was applied to XP and VX Ace but not to the VX duplicate (I-1). **Resolved:** extracted `RgssadCore.fs` — the shared parse loop, sentinel rule (DU), and bounds/negative-length guard now live once; `Xp.fs`/`Vx.fs` are thin wrappers (version byte + sentinel), so the two formats can no longer drift.

Both the C-1 build break and the 6 warnings survive because **CI's `build` step is the only place `the CLI module` is compiled with `TreatWarningsAsErrors`, and that step currently fails** — while the locally-run test executable never compiles `the CLI module` at all, so green local tests give false confidence.

**Recommended first action:** fix C-1 (`let hrFmt = Format.toString` in `the CLI module`) — it unblocks the entire CI pipeline and removes the report-mislabeling in one line.

**Order of operations:**
1. C-1 — restore the build (one line) and add a CLI smoke test that compiles `the CLI module`.
2. C-2 — add `safeJoin` containment check to MZ + VX Ace extraction; add a traversal test.
3. I-1 — mirror XP's guard into VX (or extract a shared parser); add VX tests.
4. I-4 — add `upload-artifact` steps so releases work once the build is green.
5. I-2 / I-3 — single-decrypt path; disambiguate exit codes and align docs.
6. M-6 — reconcile THEORY/README with the actual code; set `TreatWarningsAsErrors` in the `.fsproj` so local builds catch regressions.
